
# 第三次课学习记录

## 今日学习主题

今天学习了第三次课“深度学习环境配置、服务器连接与实验室资源使用”。课程围绕本地环境、实验室服务器和 AutoDL 三种实验路线展开，主要介绍了服务器连接、Conda 环境管理、PyTorch 与 CUDA 检查、NAS 资源使用、智能体 Token 安全规范、共享 GPU 使用规范以及 SASRec GPU 实验的基本流程。

## 一、三种实验环境及其适用场景

### 1. 本地环境

本地环境适合学习代码、调试小规模程序和完成 CPU 实验。当暂时无法访问校园网或实验室服务器时，可以先在本地创建独立的 Python 环境并安装 CPU 版 PyTorch。

本地电脑即使没有 NVIDIA GPU，也可以完成代码阅读、模型结构检查、数据处理和基础运行测试。此时 `torch.cuda.is_available()` 返回 `False` 是正常结果，不代表 PyTorch 安装失败。

### 2. 实验室服务器

实验室服务器是日常开展 GPU 实验和论文代码复现的主要环境。使用服务器时，每个人都应使用自己的账号、个人目录和 Conda 环境，不应修改其他用户的文件，也不能占用或结束其他用户的实验进程。

### 3. AutoDL

当实验室 GPU 资源不足、需要独立实验环境或需要指定型号的 GPU 时，可以使用 AutoDL。

创建 AutoDL 实例时，需要查看 GPU 型号、显存、地区、价格和空闲状态，并优先选择带有完整 PyTorch 环境的官方镜像。实验完成后应及时保存代码和结果，并关闭实例，避免继续计费。

三种实验环境虽然硬件条件不同，但应尽量保持代码、依赖和运行命令一致。同时要记录环境信息、执行命令和实验结果，保证实验过程能够复现。

## 二、软件工具与文件资源的分工

课程介绍了不同工具在实验过程中的主要用途。

- Windows Terminal 或 PowerShell 用于测试网络和执行 SSH 命令。
- VS Code Remote SSH 用于浏览远程服务器目录、编辑代码和运行程序。
- WinSCP 适合上传或下载少量临时文件。
- Git 用于管理代码、配置文件和 README。
- NAS 用于保存和共享数据、模型及长期课程材料。
- Conda 用于为不同项目创建相互隔离的 Python 环境。

通过本次学习，我认识到 Git、NAS 和 Conda 的作用不同。Git 主要负责代码版本管理，NAS 主要负责共享和长期保存较大的数据与模型，Conda 负责管理 Python 版本和软件依赖。三者不能相互替代，应根据文件类型和实验需求合理使用。

## 三、实验室服务器连接流程

连接实验室服务器时，应按照“先检查网络，再登录服务器”的顺序进行。

首先测试服务器的 SSH 端口是否能够访问。网络检查通过后，再使用个人用户名执行 SSH 登录。

登录服务器后，可以使用以下命令检查当前身份和所在位置：

```bash
whoami
hostname
pwd
```

其中：

- `whoami` 用于查看当前登录用户名。
- `hostname` 用于查看当前服务器名称。
- `pwd` 用于查看当前所在目录。

登录成功后，应在个人 Home 目录下创建课程文件夹，避免将个人代码、数据或环境安装到公共目录中。

课程目录可以使用以下命令创建：

```bash
mkdir -p ~/research-training/lesson03
cd ~/research-training/lesson03
```

## 四、VS Code Remote SSH 的使用

VS Code Remote SSH 可以用于连接远程服务器，并在本地 VS Code 界面中浏览、编辑和运行服务器上的代码。

使用前需要在 VS Code 中安装 `Remote - SSH` 扩展，并在 SSH 配置文件中添加服务器信息，包括服务器地址、个人用户名和端口号。

连接成功后，需要确认：

- VS Code 左下角显示远程主机名称。
- 集成终端中的 `pwd` 显示服务器上的 Linux 路径。
- 当前打开的文件夹位于个人服务器目录中。

课程强调，当 VS Code Remote SSH 连接失败时，应先确认命令行 SSH 是否能够正常登录。不能因为 VS Code 连接失败，就直接重装服务器环境。

## 五、Miniconda 与独立环境管理

Miniconda 可以为不同实验创建相互独立的 Python 环境，避免直接修改系统 Python，也可以减少不同项目之间的软件版本冲突。

在服务器安装 Miniconda 前，应先通过以下命令检查服务器架构：

```bash
uname -m
```

服务器为 Linux x86_64 架构时，应下载对应的 Linux x86_64 安装包。

安装时应注意：

- 不使用 `sudo`。
- 安装到个人目录中。
- 不覆盖其他用户的安装目录。
- 不在公共目录中安装个人环境。

课程要求为第三次课创建独立环境：

```bash
conda create -n lesson03 python=3.10 -y
conda activate lesson03
```

创建完成后，需要检查：

```bash
which python
python --version
python -m pip --version
```

检查重点包括：

- Python 路径是否属于 `lesson03` 环境。
- Python 版本是否为 3.10。
- pip 与 Python 是否来自同一个环境。

通过本次学习，我进一步理解了“一项实验对应一个独立环境”的意义。不同实验可能依赖不同版本的 Python、PyTorch 和其他软件包，使用独立环境能够减少版本冲突，也便于后续记录和复现实验。

## 六、Conda 镜像与 pip 镜像

课程说明了 Conda 镜像和 pip 镜像的区别。

- Conda 镜像用于下载 Conda 环境和 Conda 软件包。
- PyPI 镜像用于通过 pip 安装 Python 软件包。

课程统一使用清华 TUNA 镜像，pip 使用的清华镜像地址为：

```text
https://pypi.tuna.tsinghua.edu.cn/simple
```

配置镜像前，应先查看和备份原有配置。配置完成后，还需要检查镜像来源是否正确。

镜像配置需要遵守以下原则：

- 不混用多个国内镜像。
- 不使用 HTTP 镜像。
- 不关闭 SSL 验证。
- 不随意添加 `trusted-host`。
- 修改配置前先检查和备份原文件。

普通 Python 软件包可以使用清华 PyPI 镜像安装，但服务器 GPU 版 PyTorch 需要根据 CUDA 版本使用对应的专用安装地址，普通 PyPI 镜像不能替代 CUDA 专用 wheel 地址。

## 七、PyTorch、GPU 与 CUDA 的关系

课程介绍了检查 GPU 和 PyTorch 环境的基本方法。

首先使用以下命令查看服务器 GPU：

```bash
nvidia-smi
```

通过 `nvidia-smi` 可以查看：

- GPU 型号。
- 显存容量和显存占用。
- GPU 利用率。
- 当前运行的进程。
- NVIDIA 驱动信息。

在 Python 环境中，还需要使用 PyTorch 进一步检查：

```python
import torch

print(torch.__version__)
print(torch.version.cuda)
print(torch.cuda.is_available())
print(torch.cuda.get_device_name(0))
```

我理解到，`nvidia-smi` 能够看到 GPU，只能说明服务器安装了 NVIDIA 驱动并识别到 GPU，并不代表当前 Python 环境中的 PyTorch 一定能够调用 GPU。

`nvidia-smi` 显示的 CUDA Version 表示当前驱动能够支持的 CUDA 版本上限，而 PyTorch 实际使用的 CUDA 运行时需要通过 `torch.version.cuda` 检查。

在本地 CPU 环境中，`torch.cuda.is_available()` 返回 `False` 是正常结果，不表示 PyTorch 安装失败，也不需要为了得到 `True` 而安装 CUDA、cuDNN 或模拟 GPU。

## 八、暂时不能连接服务器时的处理方式

如果暂时无法连接校园网和实验室服务器，可以先完成本地 CPU 环境配置。

基本流程包括：

1. 创建 `lesson03` 独立环境。
2. 安装 Python 3.10。
3. 安装 CPU 版 PyTorch。
4. 检查 PyTorch 是否能够正常导入。
5. 检查 `torch.cuda.is_available()` 的结果。

本地 CPU 环境中，`torch.cuda.is_available()` 输出 `False` 是正常现象。

本地环境的主要作用是完成代码学习、基础调试和 CPU 测试。具备校园网和服务器访问条件后，再补充 SSH 登录、服务器 GPU 检查和 CUDA 环境验证。

## 九、AutoDL 的基本使用规范

AutoDL 适用于实验室 GPU 资源不足、需要独立环境或需要特定型号 GPU 的情况。

创建实例时，需要查看：

- GPU 型号。
- GPU 显存。
- 实例所在地区。
- 使用价格。
- 当前空闲状态。
- 系统盘和数据盘配置。
- 计费方式。

应优先选择带有完整 PyTorch 环境的官方镜像，避免重复安装 Miniconda、CUDA 和 PyTorch。

实例启动后，可以执行：

```bash
nvidia-smi
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
pwd
df -h
```

这些命令分别用于检查 GPU、PyTorch、当前目录和磁盘空间。

实验完成后，需要保存重要的代码、日志、模型和结果，并返回实例列表执行关机。

需要注意，关机、释放实例和数据保留是不同的操作，不能因为实例已关机就认为费用和数据问题已经全部处理完成。

## 十、NAS 共享文件夹的使用边界

NAS 主要用于共享课程数据、模型和长期材料。

登录 NAS 后，可以进入课程共享目录下载所需文件。使用公共文件时，需要遵守以下原则：

- 不随意删除、移动、覆盖或改名公共文件。
- 不在共享根目录中散放个人文件。
- 不确定是否可以修改时，先下载副本。
- 下载后在个人目录中解压和运行。
- 使用公共电脑后及时退出个人账号。

NAS 不应保存以下内容：

- Token。
- 密码。
- SSH 私钥。
- 含有敏感信息的配置文件。
- 大量无用的 Checkpoint。
- 重复数据。

NAS 主要用于数据、模型和长期共享材料，不能替代 Git 的代码版本管理功能。

## 十一、智能体 Token 安全规范

Token 属于个人访问凭证，只能由本人用于科研和学习，不能外传、转借、出售或公开共享。

Token 不能放在以下位置：

- Git 仓库。
- NAS 公共目录。
- Notebook。
- 代码文件。
- 截图。
- 群聊。
- 公开文档。

中转站的并发资源有限，高峰期可能需要等待。请求失败不一定代表 Token 失效，因此不能连续刷新、无限重试或同时启动大量请求。

如果发现 Token 泄露或出现异常消耗，应立即联系管理员处理。

课程还强调，智能体输出必须经过人工核验，不能代替自己的理解、分析和判断。

## 十二、SASRec GPU 实验流程

SASRec 快速实验的主要目标是验证当前环境能够调用 GPU，并跑通完整的推荐模型训练入口，而不是获得正式的论文级实验指标。

基本流程包括：

1. 获取并解压代码。
2. 激活 `lesson03` 环境。
3. 安装固定依赖。
4. 运行环境检查脚本。
5. 执行一次 SASRec 训练。
6. 观察训练日志和 GPU 进程。
7. 保存模型和输出结果。

典型命令包括：

```bash
python check_environment.py
python train.py --device cuda --epochs 1
```

运行时需要重点确认：

- 当前运行设备为 CUDA。
- 数据能够正常加载。
- 训练 Loss 能够正常输出。
- `nvidia-smi` 中出现自己的 Python 进程。
- 日志和模型正确保存到 `outputs/` 目录。

本次快速实验主要验证代码入口和运行环境，不展开 SASRec 完整原理、正式推荐指标和论文级训练结果。

## 十三、共享 GPU 使用规范

使用共享 GPU 前，必须实时执行：

```bash
nvidia-smi
```

不能根据以前的截图判断当前 GPU 是否空闲，因为显存占用和运行进程会随时发生变化。

教师指定物理 GPU 后，可以使用：

```bash
CUDA_VISIBLE_DEVICES=0 python train.py --device cuda --epochs 1
```

`CUDA_VISIBLE_DEVICES` 用于限制当前进程能够看到的物理 GPU。

当只暴露一张物理 GPU 时，程序内部通常仍将其显示为 `cuda:0`。这里的 `cuda:0` 表示当前程序可见设备中的第 0 张卡，不一定对应服务器上的物理 GPU 0。

共享 GPU 使用中需要遵守以下规范：

- 不结束其他用户的进程。
- 不暂停其他用户的进程。
- 不抢占其他用户正在使用的 GPU。
- 不确定进程归属时先询问。
- 显存不足时等待上一组实验结束。
- 实验结束后确认自己的进程已退出。
- 不留下无意义的后台进程。

## 十四、常见问题的排查顺序

课程给出了从网络、认证、工具、环境到代码的排错顺序。

### 1. 服务器无法连接

先检查是否处于校园网环境，并检查服务器 SSH 端口是否能够访问。

### 2. SSH 认证失败

检查个人用户名、密码或密钥是否正确。

### 3. VS Code 连接失败

先确认命令行 SSH 是否可以正常登录，再排查 VS Code Remote SSH 配置。

### 4. Conda 找不到

检查 Miniconda 的安装位置，并确认 Shell 初始化是否完成。

### 5. 软件包装到错误环境

使用以下命令检查：

```bash
which python
python -m pip --version
```

确认 Python 和 pip 是否属于同一个 Conda 环境。

### 6. 镜像中找不到软件包

检查镜像是否已经同步该版本。必要时可以在单次命令中临时指定其他可用来源，但不应随意修改全局配置。

### 7. PyTorch 无法使用 GPU

检查：

- PyTorch 版本。
- `torch.version.cuda`。
- `torch.cuda.is_available()`。
- 是否误装了 CPU 版 PyTorch。
- 当前 GPU 是否可用。

### 8. SASRec 运行失败

检查：

- 当前所在目录。
- 数据路径。
- 依赖是否完整。
- 运行设备是否正确。
- 命令参数是否正确。

完整的排错记录应包含：

- 问题现象。
- 当前环境。
- 执行命令。
- 完整报错。
- 已尝试的方法。
- 最终解决结果。

不能只记录“出现错误”和“已经解决”。
