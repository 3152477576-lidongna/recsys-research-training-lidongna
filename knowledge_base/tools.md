# 实验环境与远程资源使用

## 一、三种实验环境

深度学习实验通常可以在本地环境、实验室服务器和 AutoDL 云端算力平台中完成。选择哪种路线，主要取决于当前网络条件、算力需求和实验规模。

### 1. 本地环境

本地环境适合：

- 学习和阅读代码
- 调试小规模程序
- 完成 CPU 测试
- 暂时无法连接校园网或实验室服务器时使用
- 在正式使用服务器前检查代码和基础依赖

本地电脑没有 NVIDIA GPU 时，仍然可以正常安装和使用 CPU 版 PyTorch。

此时：

```python
torch.cuda.is_available()
```

返回：

```text
False
```

属于正常现象，不代表 PyTorch 安装失败，也不需要为了得到 `True` 而安装 CUDA、cuDNN 或模拟 GPU。

### 2. 实验室服务器

实验室服务器适合：

- 日常 GPU 实验
- 深度学习模型训练
- 论文代码复现
- 需要较大显存和计算资源的任务
- 多人共享算力资源

使用服务器时，应使用自己的：

- 服务器账号
- 个人目录
- Conda 环境
- 项目目录

不得修改其他用户的文件，也不能结束、暂停或抢占其他用户的实验进程。

### 3. AutoDL

AutoDL 适合：

- 实验室 GPU 暂时不足
- 需要独立实验环境
- 需要指定型号的 GPU
- 需要快速使用预配置的深度学习环境

创建实例时，需要检查：

- GPU 型号
- GPU 显存
- 实例地区
- 使用价格
- 空闲状态
- 系统盘容量
- 数据盘容量
- 计费方式
- 镜像中是否已经配置 PyTorch

应优先选择带有完整 PyTorch 环境的官方镜像，避免重复安装 Miniconda、CUDA 和 PyTorch。

### 4. 三种环境的共同原则

无论选择哪种环境，都应遵守以下原则：

- 代码尽量保持一致
- 根据设备选择 CPU 或 GPU
- 一个项目使用一个独立环境
- 记录 Python 和依赖版本
- 保存运行命令、日志和结果
- 不在代码中写入密码、私钥和 Token
- 保证实验过程可检查、可复现、可排查

## 二、不同工具与资源的分工

| 工具或资源 | 主要用途 |
|---|---|
| Terminal / PowerShell | 测试网络、执行 SSH 和系统命令 |
| VS Code Remote SSH | 浏览远程目录、编辑和运行服务器代码 |
| WinSCP | 上传和下载少量临时文件 |
| Git | 管理代码、配置文件和 README |
| NAS | 保存和共享数据、模型及长期材料 |
| Conda | 管理 Python 版本和项目依赖 |
| AutoDL | 按需使用独立 GPU 资源 |

Git、NAS 和 Conda 的作用不同：

- Git 负责代码版本管理。
- NAS 负责共享和长期保存较大的数据、模型和课程材料。
- Conda 负责 Python 版本和软件依赖管理。
- AutoDL 提供按需租用的云端 GPU 资源。

这些工具不能相互替代，应根据实验内容合理使用。

## 三、实验室服务器连接流程

服务器连接应按照以下顺序进行：

```text
检查当前网络
→ 测试 SSH 端口
→ 使用命令行 SSH 登录
→ 检查用户身份和目录
→ 创建个人课程目录
→ 使用 VS Code Remote SSH
```

### 1. 测试服务器端口

在 Windows PowerShell 中可以执行：

```powershell
Test-NetConnection <服务器地址> -Port 22
```

当结果中的：

```text
TcpTestSucceeded
```

显示为：

```text
True
```

说明当前网络可以访问服务器的 SSH 端口。

在 macOS 或 Linux 中，可以执行：

```bash
nc -vz <服务器地址> 22
```

只有端口测试成功后，才继续尝试 SSH 登录。

如果端口测试超时，应先检查：

- 当前是否连接校园网
- 是否需要学校提供的网络访问方式
- 服务器地址和端口是否正确
- 当前网络是否限制 SSH 连接

### 2. 使用 SSH 登录服务器

基本命令为：

```bash
ssh <个人用户名>@<服务器地址>
```

第一次连接时，终端可能提示是否信任服务器。核对服务器信息后输入：

```text
yes
```

再输入个人服务器密码。

输入密码时，终端通常不会显示星号或圆点，这是正常现象。

### 3. 登录后检查身份和路径

登录服务器后执行：

```bash
whoami
hostname
pwd
```

其中：

- `whoami`：查看当前登录用户名
- `hostname`：查看服务器名称
- `pwd`：查看当前所在目录

还可以执行：

```bash
ls -lah
```

查看当前目录中的文件和文件夹。

### 4. 创建个人课程目录

```bash
mkdir -p ~/research-training/lesson03
cd ~/research-training/lesson03
pwd
```

个人代码、环境、日志和实验结果应保存在个人 Home 目录中，不应放在公共目录或其他用户目录中。

## 四、VS Code Remote SSH 的使用

VS Code Remote SSH 用于在本地 VS Code 中连接远程服务器，并直接浏览、编辑和运行服务器上的代码。

### 1. 安装扩展

在 VS Code 扩展页面中搜索并安装：

```text
Remote - SSH
```

### 2. 打开 SSH 配置文件

按下：

```text
Command + Shift + P
```

Windows 中通常使用：

```text
Ctrl + Shift + P
```

在命令面板中搜索并选择：

```text
Remote-SSH: Open SSH Configuration File
```

### 3. 添加服务器配置

在 SSH 配置文件中添加：

```sshconfig
Host lab-server
    HostName <服务器地址>
    User <个人用户名>
    Port 22
```

各字段含义如下：

- `Host`：本地使用的服务器别名
- `HostName`：服务器地址
- `User`：个人服务器用户名
- `Port`：SSH 端口

服务器密码、私钥和 Token 不应写入这个配置示例或公开笔记中。

### 4. 连接远程服务器

再次打开命令面板，选择：

```text
Remote-SSH: Connect to Host
```

再选择：

```text
lab-server
```

连接成功后，在 VS Code 中打开服务器个人课程目录：

```text
~/research-training/lesson03
```

### 5. 检查是否进入远程环境

连接后需要确认：

- VS Code 左下角显示远程主机名称
- 当前打开的是服务器目录
- 集成终端中的 `pwd` 显示 Linux 服务器路径
- 当前 Python 和 Conda 来自服务器，而不是本地电脑

可以在 VS Code 集成终端中执行：

```bash
whoami
hostname
pwd
which python
```

### 6. VS Code Remote SSH 排错顺序

如果 VS Code Remote SSH 无法连接，应先检查命令行 SSH：

```bash
ssh <个人用户名>@<服务器地址>
```

正确排查顺序为：

```text
检查校园网和 SSH 端口
→ 测试命令行 SSH
→ 检查用户名和认证信息
→ 检查 VS Code Remote SSH 配置
→ 检查扩展状态
```

只有命令行 SSH 已经可以正常连接，才继续排查 VS Code。

不能因为 VS Code 连接失败，就直接重装服务器环境、Miniconda 或 Python 依赖。

## 五、AutoDL 的基本使用流程

### 1. 选择实例

创建 AutoDL 实例时，需要查看：

- GPU 型号
- GPU 显存
- 地区
- 价格
- 空闲状态
- 系统盘
- 数据盘
- 计费方式
- 官方镜像中包含的软件

应优先选择已经配置完整 PyTorch 环境的官方镜像。

### 2. 开机后检查环境

实例启动后，可以执行：

```bash
nvidia-smi
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"
pwd
df -h
```

这些命令分别用于检查：

- GPU 型号和使用状态
- PyTorch 版本和 GPU 可用性
- 当前工作目录
- 磁盘空间

### 3. 保存实验结果

实验过程中应保存：

- 项目代码
- 训练日志
- 模型文件
- 实验指标
- 运行参数
- 环境信息

重要文件不能只保存在临时系统盘中。

### 4. 使用结束后关机

实验完成后应：

1. 确认训练进程已结束。
2. 保存重要文件。
3. 返回实例列表。
4. 执行关机。
5. 检查实例状态。
6. 检查计费是否停止。

需要注意：

```text
关机
≠ 释放实例
≠ 删除数据
```

这三种操作需要分别确认。

## 六、NAS 共享文件夹的使用

NAS 主要用于保存和共享：

- 课程数据
- 模型文件
- 课程材料
- 长期实验资源
- 多人需要共同使用的大文件

### 1. 使用基本流程

1. 使用个人 NAS 账号登录。
2. 进入课程共享目录。
3. 查找需要的代码、数据或材料。
4. 将文件下载到个人目录。
5. 在个人目录中解压、修改和运行。
6. 使用公共电脑后退出个人账号。

### 2. 公共文件使用原则

使用 NAS 公共目录时，应遵守：

- 不随意删除公共文件
- 不随意移动公共文件
- 不覆盖或改名公共文件
- 不在共享根目录散放个人文件
- 不确定是否可以修改时先下载副本
- 在个人目录中完成修改和运行

### 3. 不应上传到 NAS 的内容

NAS 不应保存：

- Token
- 密码
- SSH 私钥
- 含敏感信息的配置文件
- 大量无用 Checkpoint
- 重复数据
- 与科研学习无关的大量文件

NAS 不能替代 Git 的代码版本管理功能。

## 七、智能体 Token 安全规范

Token 属于个人访问凭证，只能由本人用于科研和学习。

不得：

- 外传
- 转借
- 出售
- 公开共享
- 用于无关的大规模调用

Token 不能写入以下位置：

- Git 仓库
- NAS 公共目录
- Notebook
- Python 代码
- 配置文件
- 截图
- 群聊
- 公开文档

中转服务的并发资源有限，高峰期可能需要等待。

请求失败不一定表示 Token 已失效，因此不应：

- 连续刷新
- 无限重试
- 同时启动大量请求
- 在多个任务中重复调用

如果 Token 泄露或出现异常消耗，应立即联系管理员处理。

智能体生成的结果必须经过人工检查，不能替代自己的理解、分析和判断。

## 八、共享 GPU 使用规范

### 1. 运行前检查

每次运行实验前都应执行：

```bash
nvidia-smi
```

重点查看：

- GPU 型号
- 显存总量
- 显存占用
- GPU 利用率
- 当前运行进程
- 可用 GPU

GPU 状态会实时变化，不能根据旧截图判断当前 GPU 是否空闲。

### 2. 指定 GPU

教师指定物理 GPU 后，可以使用：

```bash
CUDA_VISIBLE_DEVICES=0 python train.py --device cuda --epochs 1
```

`CUDA_VISIBLE_DEVICES` 用于限制当前进程可以看到的物理 GPU。

如果只暴露一张 GPU，程序内部通常仍然显示：

```text
cuda:0
```

这里的 `cuda:0` 表示当前程序可见设备中的第 0 张卡，不一定对应服务器上的物理 GPU 0。

### 3. 共享资源使用边界

不得：

- 结束其他用户进程
- 暂停其他用户进程
- 抢占其他用户正在使用的 GPU
- 未经允许修改其他用户文件
- 随意使用教师未分配的 GPU

不确定进程归属时，应先询问。

显存不足时，应等待其他任务结束，或与教师和其他使用者协调。

### 4. 实验结束后的检查

实验结束后应：

- 确认训练已经停止
- 保存日志和模型
- 检查自己的进程是否退出
- 确认显存是否释放
- 不留下无意义的后台任务

## 九、常见问题的排查顺序

环境和服务器问题应按照“网络—认证—工具—环境—代码”的顺序排查。

### 1. 服务器无法连接

检查：

- 是否连接校园网
- SSH 端口是否可访问
- 服务器地址是否正确
- 当前网络是否限制 SSH

### 2. SSH 认证失败

检查：

- 用户名是否正确
- 密码是否正确
- SSH 私钥是否正确
- 是否误用了他人的账号

### 3. VS Code Remote SSH 连接失败

先检查命令行 SSH：

```bash
ssh <个人用户名>@<服务器地址>
```

命令行 SSH 成功后，再排查 VS Code 配置和扩展。

### 4. Conda 找不到

检查：

```bash
which conda
conda --version
```

服务器中还可以检查 Miniconda 的实际安装路径。

### 5. 软件包装到错误环境

检查：

```bash
which python
python -m pip --version
```

重点确认 Python 和 pip 是否来自同一个 Conda 环境。

### 6. PyTorch 无法使用 GPU

检查：

```bash
nvidia-smi
python -c "import torch; print(torch.__version__)"
python -c "import torch; print(torch.version.cuda)"
python -c "import torch; print(torch.cuda.is_available())"
```

需要确认：

- 系统是否识别 GPU
- PyTorch 是否安装正确
- PyTorch 是否为 GPU 版本
- CUDA 运行时是否匹配
- 是否误装了 CPU 版 PyTorch

### 7. 训练程序运行失败

检查：

- 当前所在目录
- 数据路径
- 依赖是否完整
- Python 环境是否正确
- 设备参数是否正确
- 命令参数是否正确
- 输出目录是否有权限

### 8. 排错记录要求

完整排错记录应包含：

- 问题现象
- 当前环境
- 执行命令
- 完整报错
- 已尝试的方法
- 最终解决结果

不能只记录：

```text
出现错误
```

或者：

```text
问题已经解决
```

而应记录完整的判断和处理过程。

## 十、实验记录与敏感信息保护

一次完整实验记录应包括：

- 使用的实验路线
- Conda 环境名称
- Python 版本
- PyTorch 版本
- GPU 型号
- CUDA 检查结果
- 运行命令
- 日志位置
- 模型和输出位置
- 遇到的问题
- 排查过程
- 最终结果

以下内容不得进入 Git、NAS、代码、笔记或截图：

- 服务器密码
- SSH 私钥
- NAS 凭证
- 智能体 Token
- 其他个人访问凭证

实验记录应满足：

```text
可复现
可检查
可排错
不包含敏感信息
```
